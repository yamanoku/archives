# yamanoku 文体スキル

yamanoku名義の記事を書く・推敲する際は、以下の文体特性に従うこと。

## 記事タイプの見極め

文体は媒体・タイプで切り替わる。まずどれかを決める。

1. **技術解説・紹介記事** — です・ます調。見出しで構造化。
2. **エッセイ・内省** — だ・である調〜口語。見出しなしか最小限。短い段落を改行で重ねる。
3. **年次振り返り・活動記録** — です・ます調ベース＋月別や項目別の箇条書き大量、リンク多用、太字でハイライト。
4. **イベントレポート・登壇告知** — です・ます調、丁寧め（「〜いたします」「ご参加ください」）。

## 共通のクセ

- **一人称は「自分」または「私」**。エッセイでは「自分」が優勢。
- **文末のバリエーション**: 「〜と思います」「〜と思っています」「〜かなと思います」「〜気がします」など、断定を避けたやわらかい推量表現を多用する。ただし主張したいことは「〜でしかない」「〜に値する」と強く言い切ることもある。
- **書き出しは前置きなしで本題か時事から入る**。例:「Model Context Protocol（MCP）、最近流行っていますね。というわけで私も〜」。挨拶や自己紹介はしない。
- **接続の口癖**: 「というわけで」「そんな訳で」「ちなみに」「一方で」「例えば」「もちろん〜ですが」。
- **リンクを非常に多用する**。参照は本文中のMarkdownリンクか、URL単独行（Zennのカード展開狙い）。人名にもX/GitHubリンクを付ける。
- **画像はGyazoが多く、altテキストを具体的・説明的に書く**（スクリーンショットの内容・数値まで記述する）。アクセシビリティへの配慮が徹底している。
- 過去の自分の記事へ「以前書いた〜をご覧ください」と自己参照する。

## 構成パターン

技術記事の典型構成:

1. 導入（2〜3段落、背景と「何をしたか/何の話か」）
2. `## ○○とは何か` — 対象の説明
3. インストール・使い方・詳細（`### `でパッケージマネージャ別などに分割）
4. 注意点・比較（類似ツールとの違いに触れることが多い）
5. `## おわりに` または `## まとめ`
6. `## 参考情報` / `## 謝辞` を付けることもある

- 「おわりに」では今後の展望・読者への呼びかけ・感謝で締める。
- 比較や状況整理にMarkdownのテーブル形式、長いリストは `<details>` で折りたたむ。

## 内容面の価値観（記事の判断基準）

本人が「[記事を書くときに気をつけること](../../../src/archives/keep-in-mind-writing-article.md)」で明言している基準:

- 既出情報の焼き直しはしない。点と点をつなげた組み合わせに価値を置く。
- 日本語で情報源がないものは書く価値がある。
- 実際に試したことを書く。試した雑感・注意点・失敗も含める。
- アクセシビリティ観点を自然に織り込む。

## やってはいけないこと

- 過度にフォーマルな硬い文章（「〜である。以上。」調の連続）
- 絵文字の乱用（本文ではほぼ使わない。引用文は除く）
- 「いかがでしたか」系の定型句を含める
- 断定だらけの説教調。主張は強くても、推量・譲歩（「もちろん〜ですが」）でバランスを取る
